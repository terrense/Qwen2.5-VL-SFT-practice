# Part A: `trt/step1_export_0323.py` 代码逐段导读

## A.1 这个文件的使命

这个文件不是“真正构建 TensorRT engine”的。

它的使命只有两个：

1. 让 TensorRT-LLM builder 执行到 ONNX 导出阶段
2. 把后续 engine 构建所需参数记录成 `build_plan_0323.json`

也就是说，它干的是：

```text
导出 + 记计划
```

而不是：

```text
导出 + 直接 build
```

这点是理解整个文件的第一前提。

### 这里在做什么

这部分是标准 Python 工程脚本开头。

#### `#!/usr/bin/env python3`

这是 shebang，用来告诉 shell：

- 这个文件应该交给 `python3` 执行

```bash
./step1_export_0323.py
```

#### `from __future__ import annotations`

这句主要服务于类型注解。

它让注解延迟求值，带来的好处通常是：

- 前向引用更稳
- 类型写法更灵活
- 一些版本兼容性更好

#### `argparse`

负责命令行参数解析。

这个文件不是只给某一个固定配置用的，它要接 TensorRT-LLM builder 的一大堆参数，所以必须用 argparse。

#### `json`

这里非常关键，因为最终 Step1 的核心产物之一就是：

- `build_plan_0323.json`

#### `Path`

用于路径处理。后面你会看到：

- `resolve()`
- `parent`
- `/` 拼接
- `mkdir()`

这些都是 `pathlib` 的典型操作。

## A.5 `make_parser_0323()` 的设计

### 对应代码块

```python
def make_parser_0323() -> argparse.ArgumentParser:
    try:
        from tensorrt_llm.tools.multimodal_builder import add_multimodal_arguments
    except Exception as exc:
        raise RuntimeError('step1_export_0323.py 依赖 tensorrt_llm / TensorRT 环境。') from exc

    parser = argparse.ArgumentParser(
        description='Step 1: export PyTorch models to ONNX and record params for a later TensorRT build.'
    )
    add_multimodal_arguments(parser)
    parser.add_argument(
        '--plan-filename',
        default=DEFAULT_PLAN_FILENAME_0323,
        help='写在 engine 目录中的构建计划文件名。',
    )
    return parser
```

### 这段为什么很重要

它体现了一个成熟工程师的设计选择：

```text
能复用底层框架参数体系，就不要自己再平行造一套
```

#### `add_multimodal_arguments(parser)`

这是 TensorRT-LLM 官方 builder 提供的参数注入函数。

它的作用是：

- 把多模态 builder 所需参数统一挂到当前 parser 上

这样做的好处：

- 参数名字和官方保持一致
- 少维护一套平行 CLI
- 后续框架升级时，兼容成本更低

#### 为什么只额外加 `--plan-filename`

因为这个参数不是 TensorRT-LLM 官方 builder 的概念，而是本项目自己的“两步式构建”概念。

也就是说：

- builder 参数复用官方
- 计划文件策略由本项目自己补充

这是非常好的“平台能力 + 项目定制”的分层方式。

## A.6 `dtype_to_string_0323()` 的协议设计

### 对应代码块

```python
def dtype_to_string_0323(dtype: Any) -> str:
    try:
        from tensorrt_llm._utils import torch_dtype_to_str
        return torch_dtype_to_str(dtype)
    except Exception:
        return str(dtype)
```

### 这段的工程意义

Step1 要把构建参数存进 JSON。

但 JSON 不擅长存这种对象：

- `torch.float16`
- `torch.bfloat16`

所以这里要做一层“序列化友好表示”转换。

#### 第一优先级

如果 TensorRT-LLM 自己提供了标准转换函数：

```python
torch_dtype_to_str(dtype)
```

那就直接复用。

#### 第二优先级

如果导入失败，就退化成：

```python
str(dtype)
```

这属于兜底策略。

这段代码体现的是一个很成熟的想法：

```text
优先复用权威实现；
不行就保证最小可用。
```

## A.8 `original_build_trt`、`plan_cache`、`initialized_paths`

### 对应代码块

```python
original_build_trt = mm_builder.build_trt_engine
plan_cache: Dict[Path, List[Dict[str, Any]]] = {}
initialized_paths: set[Path] = set()
```

### 分别是什么意思

#### `original_build_trt`

先把原始构建函数保存起来。

后面你会临时替换它，所以一定要先留一份原件，方便最后恢复。

#### `plan_cache`

这是内存里的计划缓存。

结构是：

```text
plan 文件路径 -> 对应的 build 条目列表
```

为什么是列表？

因为一个 engine 目录不一定只对应一个构建条目，可能有多个模型子部件、多个 engine。

#### `initialized_paths`

它用于判断某个计划文件是不是第一次写。

用途是：

- 第一次写时，如果旧文件存在，先删掉
- 防止旧内容和新内容混在一起

这属于“幂等性”和“避免脏状态”的思路。

## A.9 `record_build_plan(...)` 是整个 Step1 的灵魂

### 对应代码块

```python
def record_build_plan(*b_args, **b_kwargs):
    model_type = b_kwargs.get('model_type', b_args[0] if b_args else None)
    input_sizes = b_kwargs.get('input_sizes', b_args[1] if len(b_args) > 1 else None)
    onnx_dir = b_kwargs.get('onnx_dir', b_args[2] if len(b_args) > 2 else None)
    engine_dir = b_kwargs.get('engine_dir', b_args[3] if len(b_args) > 3 else None)
    ...
```

### 这段代码为什么这么写

这是为了兼容底层调用方式。

有些框架内部函数调用时：

- 用位置参数
- 有些版本用关键字参数

所以这里统一写成：

```python
b_kwargs.get('xxx', 从 b_args 中兜底取)
```

这是一种非常典型的“低耦合兼容写法”。

### 这段函数在系统上扮演什么角色

它不是普通工具函数，而是“拦截函数”。

原始 builder 流程里，本来走到某一步会调用：

```python
build_trt_engine(...)
```

现在你把这个函数换成了 `record_build_plan(...)`，就相当于：

```text
原本应该开始 build
-> 现在先别 build
-> 先把 build 所需参数记下来
```

## A.10 monkey patch 的真正含义

### 对应代码块

```python
mm_builder.build_trt_engine = record_build_plan
try:
    builder = MultimodalEngineBuilder(args)
    builder.build()
finally:
    mm_builder.build_trt_engine = original_build_trt
```

### 这是 Python 里什么级别的技巧

这是比较典型、也比较高级的运行时行为替换技巧。

#### 什么是 monkey patch

就是在程序运行中，临时替换模块、类、函数的实现。

#### 为什么这里这么做

因为你希望：

- ONNX 导出部分照常跑
- 但真正开始 build engine 时被截住

于是你把 `build_trt_engine` 改成了一个“只记录，不构建”的版本。

#### 为什么 `finally` 不能省

因为 monkey patch 是全局副作用。

如果不恢复原函数：

- 当前进程后续逻辑仍然会使用被替换版本
- 这很容易让调试变成灾难

这也是为什么大厂工程里，任何 patch / hook / override，都应该有明确的恢复路径。

## B.5 `normalize_dtype_0323()`：从字符串恢复 dtype

### 对应代码块

```python
def normalize_dtype_0323(value: Any) -> torch.dtype:
    if value is None:
        return torch.float16
    if isinstance(value, str):
        lowered = value.lower().replace('torch.', '')
        mapping = {
            'float16': torch.float16,
            'half': torch.float16,
            'fp16': torch.float16,
            'bfloat16': torch.bfloat16,
            'bf16': torch.bfloat16,
            'float32': torch.float32,
            'fp32': torch.float32,
        }
        ...
```

### 为什么这段很重要

Step1 为了写 JSON，把 dtype 存成了字符串。

但 Step2 真正构建 engine 时，需要的不是字符串，而是：

- `torch.float16`
- `torch.bfloat16`

所以这里就是一个典型的：

```text
协议层表示 -> 执行层表示
```

### 为什么同时兼容 `float16`、`fp16`、`half`

这是为了提升兼容性。

因为不同工具或不同日志输出，可能会写成：

- `float16`
- `fp16`
- `torch.float16`
- `half`

成熟工程代码不能假设外部输入永远只有一种字符串形式。

## B.6 `load_plan_0323()` 的意义

### 对应代码块

```python
def load_plan_0323(plan_path: Path) -> List[Dict[str, Any]]:
    if not plan_path.exists():
        raise FileNotFoundError(f'Plan file not found: {plan_path}')
    data = json.loads(plan_path.read_text())
    if not isinstance(data, list):
        raise ValueError('Plan file must contain a list of build entries')
    return data
```

### 这是在做什么

这不是简单“读文件”，而是在做协议校验。

校验点有两个：

1. 文件是否存在
2. JSON 结构是否符合预期

这里体现的是一种很重要的工程意识：

```text
不要盲信中间产物
```

中间产物可能因为：

- 上游写坏
- 手动编辑
- 脚本中断
- 文件损坏

而变得不可用。

所以 Step2 开头就做严格校验是很合理的。

## B.7 `main()` 的构建执行逻辑

### 对应代码块

```python
from tensorrt_llm.tools.multimodal_builder import build_trt_engine
...
plan_path = Path(args.plan_path).expanduser().resolve() if args.plan_path else (Path(args.output_dir).expanduser().resolve() / DEFAULT_PLAN_FILENAME_0323)
plan_entries = load_plan_0323(plan_path)
...
for idx, entry in enumerate(plan_entries, start=1):
    dtype = normalize_dtype_0323(entry.get('dtype'))
    delete_onnx = entry.get('delete_onnx', True) and not args.keep_onnx
    build_trt_engine(
        model_type=entry['model_type'],
        input_sizes=entry['input_sizes'],
        onnx_dir=entry['onnx_dir'],
        engine_dir=entry['engine_dir'],
        max_batch_size=entry['max_batch_size'],
        dtype=dtype,
        model_params=entry.get('model_params'),
        onnx_name=entry.get('onnx_name', 'model.onnx'),
        engine_name=entry.get('engine_name', 'model.engine'),
        delete_onnx=delete_onnx,
    )
```

### 怎么理解这段

#### `build_trt_engine`

Step2 真正依赖的核心只有这个函数。

这说明 Step2 不是一个复杂 orchestrator，而是一个很纯的执行器。

#### `plan_path = A if condition else B`

这是一种 Python 三元表达式。

含义是：

- 如果用户给了 `--plan-path`，就用它
- 否则就在 `--output-dir` 下面拼默认计划文件名

#### `enumerate(plan_entries, start=1)`

`enumerate` 会把列表变成：

- 索引
- 元素

这里加 `start=1` 是为了日志更符合人类阅读习惯：

```text
[1/3]
[2/3]
[3/3]
```

而不是从 0 开始。

#### `delete_onnx`

计划文件里可能写着：

- 构建完删不删 ONNX

但如果用户显式传了：

```bash
--keep-onnx
```

那运行时应覆盖这个行为。

这体现的是：

- 静态计划
- 运行时调试开关

之间的组合关系。

# Part C: `deploy/deploy_api_0323.py` 代码逐段导读

## C.1 这个文件的使命

它的使命不是训练，不是导出，不是量化。

它的使命是：

```text
把本地模型封装成一个可通过 HTTP 调用的 OpenAI-compatible 多模态服务
```

外部系统以后不需要知道你内部是：

- HuggingFace
- PEFT
- merge 模型
- base + LoRA

它只需要知道：

- 发一个 HTTP 请求
- 路径是 `/v1/chat/completions`
- 按约定格式传图和文字

### 这些类为什么非常关键


- 服务端应该怎么理解

所以 `content` 不能只允许一种形态。

这就是为什么这里写：

```python
Union[str, List[MessageContent0323]]
```

## C.7 `load_model_and_processor_0323()`：双模式加载设计

```python
def load_model_and_processor_0323(args: argparse.Namespace):
    if args.mode == 'merged':
        ...
        model = Qwen2_5_VLForConditionalGeneration.from_pretrained(...)
        processor = AutoProcessor.from_pretrained(...)
```

### 这段是整个服务的模型加载核心

#### 模式一：`merged`

直接从 merge 后目录加载完整模型。

优点：

- 结构最简单
- 部署最直接
- 更适合作为后链最终形态

#### 模式二：`lora`

先加载 base model，再叠 LoRA adapter。

优点：

- 更接近训练态
- 更适合保留 adapter 形式的实验环境

### 为什么 `processor` 也跟着切换

因为不同模型目录可能自带不同 processor 配置。

在 `merged` 模式下，从 merged 模型目录拿 processor 是最自然的。

在 `lora` 模式下，processor 仍回到 base model。

## C.8 `extract_image_and_text_0323()`：协议拆包层

### 对应代码块

```python
def extract_image_and_text_0323(request: ChatCompletionRequest0323):
    image_obj, text_obj = None, None
    for msg in request.messages:
        if isinstance(msg.content, list):
            for item in msg.content:
                if item.type == 'image' and item.image:
                    if 'base64' in item.image:
                        img_data = base64.b64decode(item.image['base64'])
                        image_obj = Image.open(io.BytesIO(img_data)).convert('RGB')
                    elif 'image' in item.image:
                        image_obj = Image.open(item.image['image']).convert('RGB')
                elif item.type == 'text':
                    text_obj = item.text
    return image_obj, text_obj
```

### 这段代码为什么重要

这是“外部协议 -> 内部对象”的第一层转换。

它做了两件事：

1. 识别图像输入
2. 识别文本输入

### 为什么支持 base64 和本地路径两种方式

这是为了兼顾两类使用场景：

#### base64

适合：

- 前端上传
- Web 客户端调用
- 远程请求

#### 本地路径

适合：

- 本机调试
- 内部脚本联调
- 快速 demo

### `Image.open(io.BytesIO(...)).convert('RGB')` 在做什么

这是标准的内存图片解码流程：

1. base64 -> bytes
2. bytes -> BytesIO
3. BytesIO -> PIL.Image
4. 统一转成 RGB

这样后续视觉处理更稳定。

## C.11 推理和输出裁剪

### 对应代码块

```python
with torch.no_grad():
    generated_ids = MODEL_0323.generate(**inputs, max_new_tokens=request.max_tokens or MAX_NEW_TOKENS_0323)

generated_ids_trimmed = [
    out_ids[len(in_ids):] for in_ids, out_ids in zip(inputs.input_ids, generated_ids)
]
output_text = PROCESSOR_0323.batch_decode(
    generated_ids_trimmed,
    skip_special_tokens=True,
    clean_up_tokenization_spaces=False,
)[0]
```

### 逐段解释

#### `torch.no_grad()`

推理时关闭梯度，减少显存和计算开销。

#### `generate(...)`

这是 HuggingFace 生成式模型的标准推理接口。

#### 为什么要裁掉前面的 token

因为 `generate()` 返回的是：

- 输入 prompt token
- 新生成 token

而服务想返回的只是模型的新回答。

所以要做：

```python
out_ids[len(in_ids):]
```

这属于生成式服务中的标准写法。
