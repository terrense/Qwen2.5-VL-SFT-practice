这份学习手册不是执行清单，而是“理解清单”。

它面向这样的读者：

- 已经拿到 `qwen25vl_sft_chain_0323/` 目录

- 但还没有完全吃透这条 `SFT -> merge -> prune -> quantize -> TRT -> deploy` 工程链

本文重点回答 4 类问题：

1. 这条 chain 从工程上到底想解决什么问题
2. 每个阶段在整个系统中的位置是什么
3. 关键脚本的 Python 语法、实现技巧、工程意图是什么
4. `trt/step1_export_0323.py`、`trt/step2_build_0323.py`、`deploy/deploy_api_0323.py` 到底是怎么工作的

## 1. 先把全局图景建立起来

这套项目不是只有 SFT，它实际上是一条完整的“模型研究到模型部署”的通路。

### 1.1 三层架构

可以把整个目录里的代码分成三层：

#### 第一层：训练验证层

```text
baseline eval
-> LoRA SFT
-> LoRA eval
-> single-image inference
```

这一层的目标是：

- 确认数据可读
- 确认模型可正常推理
- 确认 LoRA 可以训练
- 确认微调后指标可回收

#### 第二层：模型工程层

```text
merge
-> prune
-> pruned eval
-> quantize
```

这一层的目标是：

- 把训练产物变成后续链路可消费的标准模型形式
- 逐步向“更适合部署”的模型形态推进

#### 第三层：部署加速层

```text
ONNX export
-> TensorRT engine build
-> API deploy
```

这一层的目标是：

- 把模型从“Python 里能跑”变成“更适合线上服务和高性能推理”

## 2. 一句话理解整条 chain

如果你想快速抓住本质，可以先记住下面这句：

```text
LoRA 是训练时的轻量增量表示；
merge 是把增量还原成完整模型；
prune 和 quantize 是把完整模型继续往更可部署方向加工；
TensorRT 是把模型变成更贴硬件的推理引擎；
API deploy 是把模型能力用标准协议暴露给外部系统。
```

这句几乎可以概括整条后链路的工程意义。

## 3. 为什么训练之后不能直接“结束”

很多初学者会以为：

```text
训练完了 -> 有 checkpoint -> 结束
```

但真正的工业环境往往不是这样。

### 3.1 训练产物和部署产物不是一个东西

训练阶段产出的东西通常偏研究态：

- LoRA adapter
- checkpoint
- trainer 日志
- HuggingFace 训练配置

而部署阶段更关心的是：

- 可独立加载的完整模型目录
- 更省显存的量化表示
- 更高性能的引擎文件
- 更统一的在线服务接口

所以训练结束以后，通常还要继续做：

1. merge
2. prune
3. quantize
4. ONNX / TensorRT
5. API 封装

### 3.2 工业系统和研究脚本的关注点不同

研究脚本关心的是：

- 指标能不能提升
- loss 降不降
- 某个任务上有没有收益

部署系统关心的是：

- 模型是否独立可加载
- 延迟能否接受
- 显存是否可控
- 是否方便被业务系统调用

这就是为什么你目录里会同时出现：

- `train_lora_full_0323.py`
- `post_sft/merge_lora_0323.py`
- `quantization/quantize_0323.py`
- `trt/step1_export_0323.py`
- `deploy/deploy_api_0323.py`

## 4. LoRA 是什么，为什么后面还要 merge

### 4.1 LoRA 的本质

LoRA 不是重新训练整个大模型，而是在原始大权重矩阵外面，挂一个低秩增量更新。

它的直觉可以写成：

```text
W_new = W_base + Delta_W
```

其中：

```text
Delta_W = B @ A * scale
```

这里：

- `W_base` 是原始大模型权重
- `A` 和 `B` 是低秩矩阵
- `Delta_W` 是一个低参数量、低显存成本的增量

LoRA 的价值在于：

- 训练便宜
- 显存友好
- 适合大模型微调

### 4.2 为什么训练后不总是保留 LoRA 形式

LoRA 在训练和研究阶段很舒服，但在后处理和部署阶段不一定最方便。

原因是：

- 很多工具链更喜欢完整模型目录
- 结构化剪枝一般要直接操作完整权重矩阵
- 量化往往更适合基于完整模型做
- TensorRT / 某些部署系统不喜欢“基座模型 + adapter”双件套

所以要做：

```text
merge_lora_0323.py
```

也就是把 LoRA 增量真正加回基座矩阵，得到一个完整模型。

## 5. `post_sft/merge_lora_0323.py` 深度讲解

这个脚本的定位是：

```text
把“基座模型 + LoRA adapter”
转成
“一个完整 HuggingFace 模型目录”
```

### 5.1 它在系统中的位置

它是整个后链路的起点。

没有 merge，就很难顺利进入下面这些步骤：

- 结构化剪枝
- 量化
- TensorRT 导出
- 直接部署

### 5.2 关键代码结构

核心逻辑分 4 步：

1. 加载 base model
2. 加载 LoRA adapter
3. 执行 `merge_and_unload()`
4. 保存完整模型 + tokenizer + processor

### 5.3 `sys.path` 注入在做什么

脚本里有这样一段：

```python
PROJECT_ROOT_0323 = Path(__file__).resolve().parents[1]
if str(PROJECT_ROOT_0323) not in sys.path:
    sys.path.insert(0, str(PROJECT_ROOT_0323))
```

这是因为脚本位于 `post_sft/` 子目录。

如果直接运行：

```bash
python post_sft/merge_lora_0323.py
```

Python 默认不一定能自动找到项目根目录里的 `experiment_config_0323.py`。

所以这里手动把项目根目录加入模块搜索路径。

#### 这属于什么知识点

这是 Python 的模块搜索路径机制：

- `sys.path` 是一个列表
- Python 会按这个列表中的目录顺序去找模块
- `insert(0, ...)` 表示优先把项目根目录放在最前面

### 5.4 为什么用 `Path`

现代 Python 工程更推荐 `pathlib.Path`，原因是它比字符串路径更强。

例如：

- `exists()`
- `resolve()`
- `mkdir()`
- `/` 拼接路径

比起手写：

```python
os.path.join(...)
```

它更直观，也更不容易出错。

### 5.5 `argparse` 在这里的作用

脚本里通过 `argparse.ArgumentParser(...)` 定义命令行参数。

它解决的是一个非常工程化的问题：

- 同一份脚本，允许默认跑
- 也允许用户改 base model 路径、LoRA 路径、输出目录、dtype、device_map

所以它不是“多余的花哨参数”，而是提高脚本复用性和可维护性。

### 5.6 `Namespace` 是什么

`parse_args()` 返回的不是 dict，而是 `argparse.Namespace`。

所以访问方式是：

```python
args.base_model
args.lora_checkpoint
```

而不是：

```python
args["base_model"]
```

### 5.7 `device_map` 为什么重要

这是 HuggingFace 大模型加载时非常关键的参数。

它控制模型放在哪些设备上。

常见用法：

- `auto`
  让 HF / accelerate 自动分配设备
- `"cuda:0"`
  强制全模型上单卡
- 字典映射
  做更细粒度模块级设备分配

对 32B 模型来说，这不是细枝末节，而是能不能加载成功的关键。

### 5.8 `merge_and_unload()` 是真核心

这一步完成了两个动作：

1. 把 LoRA 增量真正加回原始权重矩阵
2. 去掉 PEFT 包装层，返回更普通的模型对象

你可以把它理解成：

```text
研究态双件套
-> 部署态完整模型
```

### 5.9 为什么还要保存 tokenizer 和 processor

只保存模型权重不够。

因为推理不仅需要模型，还需要知道：

- 文本怎么切 token
- 图片怎么预处理
- 多模态输入怎么组织

所以 merge 后目录必须尽量自包含：

- 模型权重
- tokenizer
- processor

否则后面虽然“有模型”，但还是没法独立推理。

## 6. `post_sft/prune_2to4_0323.py` 深度讲解

这个脚本负责：

```text
对 merge 后完整模型做 2:4 结构化剪枝
```

### 6.1 什么是 2:4 structured pruning

这不是普通“把 50% 权重随机清零”。

它要求：

- 每 4 个权重形成一个 group
- 每个 group 里只允许保留 2 个
- 另外 2 个必须变成 0

这是一种带规则的结构化稀疏模式。

### 6.2 为什么结构化稀疏比普通稀疏更有工程意义

普通非结构化稀疏的问题是：

- 虽然参数看起来少了
- 但硬件不一定能很好利用这些零

结构化稀疏的价值在于：

- 模式规则
- 更容易被编译器 / kernel 识别
- 更接近真正可加速的稀疏推理

所以工业部署更偏爱结构化稀疏，而不是纯学术意义上的任意稀疏。

### 6.3 为什么重点剪 `Linear` 层

因为：

- 注意力投影层本质上是线性层
- MLP 投影层本质上也是线性层
- 它们占大模型大部分参数量和算力

所以你脚本里的目标模块：

- `q_proj`
- `k_proj`
- `v_proj`
- `o_proj`
- `gate_proj`
- `up_proj`
- `down_proj`

本质上对应的都是关键线性层。

### 6.4 `permute -> reshape(-1, 4) -> topk -> mask` 的数学意义

这部分是整个脚本最重要的算法实现。

#### 第一步：把目标维移到最后

这是为了方便统一做 4 元分组。

如果不先 `permute`，不同层的权重张量在不同维度上做 grouping 会很难统一处理。

#### 第二步：`reshape(-1, 4)`

这一步表示：

- 前面所有维度压平
- 最后一维按 4 个一组切开

于是张量变成一个二维矩阵，每一行就是一个 4 元组。

#### 第三步：`abs() + topk(k=2)`

剪枝时通常关心“权重强度”，而不是正负号。

所以要先取绝对值，再取最大 2 个位置。

#### 第四步：构造 mask

mask 是布尔张量：

- `True` 表示保留
- `False` 表示清零

#### 第五步：原地乘零

通过：

```python
view.mul_(mask.to(view.dtype))
```

把不保留的位置变成 0。

### 6.5 `scatter_` 是什么

这是 PyTorch 很经典的“按索引写值”操作。

在这里：

- 先有一个全 False 的 mask
- 再根据 `topk` 找到的位置，用 `scatter_` 把对应位置改成 True

这是非常高效和典型的张量写法。

### 6.6 为什么是原地操作 `_`

PyTorch 里名字以 `_` 结尾的方法通常表示原地修改。

比如：

- `scatter_`
- `mul_`

原地操作的特点是：

- 不额外申请新张量
- 更节省显存
- 但必须明确知道自己在改什么

在这种“离线模型后处理”脚本里，原地改权重是合理的，因为目标就是永久改变权重。

### 6.7 `dry-run` 有什么意义

不是所有脚本都应该“一上来就写盘”。

`dry-run` 的价值是：

- 先看看哪些层会被处理
- 看看有没有层因为维度不能被 4 整除而被跳过
- 先做策略验证，再真正写模型

这属于典型的工程稳健性设计。

## 7. `post_sft/eval_pruned_0323.py` 深度讲解

这个脚本的定位是：

```text
评测 merge 后或 prune 后的完整模型目录
```

### 7.1 它和 LoRA 评测脚本有什么本质区别

核心区别在于“模型形态”：

- `eval_lora_0323.py`
  评测的是 `base + adapter`
- `eval_pruned_0323.py`
  评测的是已经导出的完整模型目录

这不是小差别，而是整个模型生命周期不同阶段的差别。

### 7.2 为什么后处理后一定要重新评测

因为后处理操作都可能引入精度损失：

- merge 理论上不应改变功能，但实现上仍建议验证
- prune 可能掉点
- quantize 更可能掉点

所以正确流程永远是：

```text
做一次模型变换
-> 重新评测
-> 再决定是否继续往下走
```

### 7.3 `strip_code_fences_0323()` 为什么存在

大模型输出经常不是纯 JSON，而是：

```text
```json
{...}
```
```

但 `json.loads()` 要求输入是严格 JSON 文本。

所以要先去掉 markdown fence。

这类函数是 LLM 工程里非常常见的“防御式清洗层”。

### 7.4 `defaultdict` 为什么适合做混淆矩阵统计

因为混淆矩阵本质上是：

```text
confusion[true_label][pred_label] += 1
```

如果用普通 dict，你需要频繁判断 key 存不存在。  
而 `defaultdict(lambda: defaultdict(int))` 可以让你直接 `+= 1`。

这既简洁，也降低 bug 风险。

### 7.5 为什么最后还要 `_to_dict`

因为：

- `defaultdict` 适合统计阶段
- 但最终落 JSON 时，普通 dict 更稳定、更通用

这也是典型工程套路：

```text
内部用方便的数据结构
输出时转成通用数据结构
```

### 7.6 为什么要同时输出 `metrics` 和 `detailed_metrics`

这是为了兼顾两种使用场景：

- 研究视角：看总体指标
- 排障视角：看单字段详细报告和混淆矩阵

这也是实验工程化的一个标志：

不是只报一个 accuracy，而是把分析链路留完整。

## 8. `quantization/quantize_0323.py` 深度讲解

这是从“完整模型”迈向“部署友好模型”的关键一步。

### 8.1 量化到底是什么

量化本质上是数值表示压缩。

例如把权重从：

- FP32
- FP16
- BF16

压缩到：

- INT8
- INT4
- FP8

### 8.2 为什么量化有价值

量化的核心收益有：

- 模型更小
- 显存占用更低
- 带宽压力更小
- 推理速度更快
- 更适合硬件部署

### 8.3 为什么量化可能伤精度

因为数值表示能力下降了。

简单说：

- 原本一个连续精细的浮点空间
- 现在被压缩到更粗的离散空间

这会带来：

- 权重误差
- 激活误差
- 输出分布变化

所以量化永远是“效率-精度权衡”。

### 8.4 `int4_awq` 和 `fp8` 是什么

#### `int4_awq`

你可以先把它理解成一种“更聪明的 4bit 权重量化方法”。

它不是最朴素的 int4，而是更强调：

- 保护关键权重
- 提高低比特量化的可用性

#### `fp8`

FP8 还是浮点数，只是精度更低。

它相对 int4 的特点通常是：

- 精度更稳一些
- 表示能力更像浮点
- 压缩比未必像 int4 那么激进

### 8.5 为什么量化需要校准数据

因为量化不只是“把权重改成低比特”，还需要知道：

- 激活大概分布在哪个范围
- 缩放因子应该怎么选
- 哪些通道更敏感

所以要让系统看一批真实样本，估计合适的量化区间。

这就是 calibration。

### 8.6 为什么这里用 `cnn_dailymail`

因为原始参考链路就这么做，它的主要定位是：

- 打通工程链
- 给 TensorRT-LLM 导出一个标准量化流程

从“业务最优”角度，最理想的做法当然是：

- 用更贴近你真实多模态任务的图文样本做校准

但从“先把链打通”的角度，这里沿用原链是合理的。

### 8.7 `DataLoader` 在这里扮演什么角色

`DataLoader` 把 calibration 数据组织成迭代器。

量化函数一般希望收到的是：

- 一批一批的输入
- 能稳定遍历
- 不需要一次性全塞进显存

这就是 `DataLoader` 的意义。

### 8.8 为什么 TensorRT-LLM 相关 import 放在 `main()` 里

这是很成熟的工程写法。

目的：

- 没有部署环境时，import 这个脚本本身不会立刻炸
- 只有真正执行量化时，才强依赖 `tensorrt_llm`

这让训练环境和部署环境可以相对隔离。

## 9. `trt` 到底是什么：从原理到工程

这是你最关心的部分之一。

### 9.1 `trt` 的全称

`trt` 基本就是 `TensorRT`。

### 9.2 TensorRT 的本质

TensorRT 不是训练框架，而是推理优化引擎。

它做的事可以概括为：

1. 接受一个模型计算图
2. 针对目标硬件做图优化
3. 选择更优 kernel
4. 规划内存布局
5. 生成最终可执行的推理 engine

它更像编译器 / 引擎，而不是研究框架。

### 9.3 ONNX 的角色

ONNX 可以看作中间交换格式。

链路通常是：

```text
PyTorch/HF
-> ONNX
-> TensorRT
-> engine
```

你可以把 ONNX 理解成：

- 更统一的图表示
- 便于不同后端接手

### 9.4 `.engine` 是什么

`.engine` 是 TensorRT 针对特定硬件、特定配置编译出来的执行产物。

它不是普通 checkpoint。

它更接近：

- 一个针对 GPU 优化过的推理二进制
- 一个“可执行推理计划”

所以：

- checkpoint 更适合研究和跨平台流转
- engine 更适合特定机器上的高性能推理

## 10. `trt/step1_export_0323.py` 讲义版彻底拆解

这个文件是整条部署链里最有“框架拦截”味道的代码。

### 10.1 Step1 的目标是什么

一句话：

```text
导出 ONNX，但不真正 build engine；
只把未来 build 所需参数记成一个计划文件。
```

### 10.2 为什么不在 Step1 直接 build

这是工业上很常见的分阶段设计。

好处：

- ONNX 导出失败和 engine 构建失败分离
- 构建更容易复用导出结果
- 新进程 build 更稳
- 更利于流水线和批处理

### 10.3 `make_parser_0323()` 为什么值得学

这个函数不是重新定义一套 TensorRT 参数，而是复用：

```python
add_multimodal_arguments(parser)
```

这说明作者的工程思路是：

- 尽量复用官方 builder 参数体系
- 降低自己维护参数协议的成本
- 避免和底层框架接口漂移

这就是大厂里常说的：

```text
尽量复用下层平台接口，不在上层重复造一套平行协议
```

### 10.4 monkey patch 是这里最大的知识点

代码核心：

```python
original_build_trt = mm_builder.build_trt_engine
...
mm_builder.build_trt_engine = record_build_plan
try:
    builder = MultimodalEngineBuilder(args)
    builder.build()
finally:
    mm_builder.build_trt_engine = original_build_trt
```

#### 它到底在做什么

`builder.build()` 内部原本会调官方的 `build_trt_engine(...)`。

现在我们把那个函数临时替换成了自己的 `record_build_plan(...)`。

于是：

- builder 仍然照常执行
- ONNX 仍然照常导出
- 一旦走到 build engine 那一步
- 就被自己的函数拦截
- 不 build，只记计划

#### 为什么这招成立

因为 Python 里：

- 函数是对象
- 模块属性也可以在运行时改写

这就是 monkey patch 的原理。

#### 为什么 `finally` 很关键

因为 monkey patch 是全局副作用。

如果不恢复原函数：

- 当前进程后面的代码都会拿到被篡改过的函数
- 非常容易产生状态污染和难查 bug

这就是为什么成熟工程代码里，凡是有 patch / hook / override，几乎都要配合 `try/finally`。

### 10.5 `record_build_plan()` 为什么同时兼容位置参数和关键字参数

这里写法是：

```python
model_type = b_kwargs.get('model_type', b_args[0] if b_args else None)
```

这是为了兼容底层 builder 的调用方式变化。

也就是说：

- 如果底层传的是关键字参数，优先从 `b_kwargs` 取
- 如果底层传的是位置参数，再从 `b_args` 取

这是一种非常实用的“弱耦合兼容写法”。

### 10.6 `plan_cache` 和 `build_plan_0323.json` 的设计意义

Step1 的输出不是 engine，而是计划单。

计划单里记录的是：

- model_type
- input_sizes
- onnx_dir
- engine_dir
- max_batch_size
- dtype
- model_params
- onnx_name
- engine_name
- delete_onnx

也就是说，Step1 不只是在“导出图”，而是在“冻结一次构建配置”。

这在工程上特别重要，因为它把：

```text
导出时使用的参数
和
构建时使用的参数
```

强绑定在一起，避免人工回忆或手工复制出错。

### 10.7 为什么 `dtype` 要先转成字符串

因为 JSON 不能优雅地直接保存 `torch.float16` 这种对象。

所以 Step1 做了：

```text
torch.dtype -> string
```

而 Step2 再做：

```text
string -> torch.dtype
```

这就是标准的序列化/反序列化思路。

### 10.8 从软件架构视角，Step1 是什么

Step1 其实像一个“构建任务编排器”或“任务生成器”。

它负责：

- 整理 build 所需上下文
- 产出规范化任务描述

而不是自己承担最终执行。

## 11. `trt/step2_build_0323.py` 讲义版彻底拆解

如果说 Step1 是“生成工单”，Step2 就是“消费工单”。

### 11.1 Step2 的目标

一句话：

```text
读 Step1 产出的 build_plan_0323.json，
按计划逐条真正构建 TensorRT engine。
```

### 11.2 它为什么比 Step1 更“纯”

因为 Step2 不再负责：

- 参数推断
- builder 参数收集
- ONNX 导出

它只负责：

1. 读取 plan
2. 恢复 dtype
3. 调 `build_trt_engine(...)`

这就是典型执行器风格。

### 11.3 `parse_args_0323()` 的双入口设计

Step2 支持：

- `--plan-path`
- `--output-dir`

为什么这样设计合理：

- 已知计划文件位置时，直接指定文件最方便
- 只知道 engine 输出目录时，也可以直接从目录里找默认文件名

这种 CLI 设计体现了“用户使用场景友好”。

### 11.4 `normalize_dtype_0323()` 是什么类型的代码

这是典型的协议恢复层。

Step1 把 dtype 存成字符串：

- `float16`
- `bf16`
- `torch.float16`

Step2 必须把它恢复成真正的 `torch.dtype` 才能传给 build 函数。

这类逻辑在大型系统中非常常见：

```text
外部表示（可存储/可传输）
<-> 内部表示（可执行/可计算）
```

### 11.5 为什么 `load_plan_0323()` 要校验是 list

因为计划文件不是任意 JSON，而是“有明确结构约束的协议文件”。

如果读出来不是列表：

- 说明 plan 已损坏
- 或者格式不符合预期

这类结构校验是成熟工程代码的重要标志：

```text
不要盲信输入
```

### 11.6 `keep-onnx` 的意义

有些时候你想保留 ONNX：

- 做排障
- 做对比
- 反复试不同 build 参数

有些时候你不想保留：

- 节省磁盘
- ONNX 只是中间产物

所以 `--keep-onnx` 是一个非常实用的调试开关。

### 11.7 从架构视角，Step2 是什么

它非常像一个“任务执行器”：

- 输入是任务描述 JSON
- 输出是 engine
- 自己不做复杂业务决策

这类角色在很多大厂系统中都非常常见。

## 12. `deploy/deploy_api_0323.py` 彻底讲透

这是你要的另一个重点。

这份脚本本质上不是算法脚本，而是 Web 服务脚本。

### 12.1 它在系统中的作用

它解决的问题是：

```text
如何把本地模型能力暴露给外部系统调用
```

也就是：

- 前端
- 标注系统
- 业务服务
- 自动化客户端

都不需要知道你内部是 HuggingFace 还是 LoRA，只需要调一个标准 HTTP 接口。

### 12.2 为什么要做成 OpenAI-compatible

因为 OpenAI Chat Completions 协议已经成为事实标准。

它的好处是：

- 很多客户端天然支持
- 前后端更容易对接
- 接口抽象稳定

所以即使底层不是 OpenAI 模型，也完全可以复用这套协议外壳。

这就是“协议兼容层”的价值。

### 12.3 它是不是 vLLM 服务

严格说，不是。

它更准确的描述是：

```text
FastAPI + HuggingFace/PEFT + OpenAI-compatible protocol adapter
```

也就是说：

- 协议像 OpenAI
- 底层推理还是 HF `generate()`

这类服务非常适合：

- 内部验证
- Demo 服务
- 统一调用协议

但如果追求极致吞吐，通常还会继续换更高性能推理后端。

## 13. `deploy_api_0323.py` 的 Python / Web 语法知识点

### 13.1 FastAPI 是什么

FastAPI 是 Python Web 框架，专长是：

- 快速定义 API
- 自动做请求校验
- 自动生成接口 schema
- 异常和路由处理相对现代化

### 13.2 Pydantic 模型在这里为什么重要

这些类：

- `MessageContent0323`
- `ChatMessage0323`
- `ChatCompletionRequest0323`
- `ChoiceMessage0323`
- `Choice0323`
- `ChatCompletionResponse0323`

不是可有可无的“样板代码”，而是：

- 请求结构定义
- 响应结构定义
- 数据校验规则
- 文档和 schema 的基础

你可以把它们理解成：

```text
接口的数据合同
```

### 13.3 `Union[str, List[MessageContent0323]]` 是什么意思

意思是：

`content` 字段允许两种形态：

- 纯字符串
- 一个由多个结构化内容项组成的列表

这正是多模态协议适配的关键点之一，因为：

- 有的消息只是文本
- 有的消息同时带图和文

### 13.4 `@app.post('/v1/chat/completions')` 是什么

这是 Python 装饰器。

它的语义是：

- 这个函数是一个 POST 接口处理函数
- 路径是 `/v1/chat/completions`

也就是说，当 HTTP 请求打到这个路径时，FastAPI 会调用这个函数。

### 13.5 `response_model=...` 的意义

它会让 FastAPI 用 Pydantic 对返回结果做规范化和校验。

好处：

- 输出格式更稳定
- 更容易保证接口一致性
- 降低返回脏结构的风险

## 14. `deploy_api_0323.py` 的系统设计讲解

### 14.1 服务启动前发生了什么

`main()` 里做了这几件事：

1. 解析启动参数
2. 加载模型和 processor
3. 把模型切到 eval 模式
4. 启动 Uvicorn 服务

这是很标准的“服务启动生命周期”。

### 14.2 为什么要支持 `merged` 和 `lora` 两种模式

因为工程里常常有两类部署诉求：

#### 模式一：`merged`

优点：

- 结构最简单
- 后续依赖少
- 更像标准独立模型目录

#### 模式二：`lora`

优点：

- 保留 base + adapter 组合形式
- 对实验态更灵活

所以提供双模式是合理的，不是多余复杂化。

### 14.3 `extract_image_and_text_0323()` 在做什么

这其实是协议适配层的核心函数之一。

它把外部请求里的多模态输入拆成：

- 图片对象
- 文本 prompt

支持两种图片来源：

- base64
- 本地文件路径

这让服务既能接浏览器上传结果，也能接本地调试输入。

### 14.4 为什么接口层还要再重组 `messages`

因为 OpenAI 协议格式和 Qwen-VL 处理器格式不是一回事。

外部是：

```text
OpenAI-like messages
```

内部模型需要的是：

```text
Qwen-VL structured messages
```

所以服务端还要做一次翻译：

```text
外部协议
-> 内部 message 结构
-> processor / process_vision_info
-> generate
-> 外部协议响应
```

这就是典型的 adapter layer。

### 14.5 为什么还要 `processor.apply_chat_template(...)`

因为模型并不是直接吃自然语言字符串，它往往有一套约定好的对话模板。

`apply_chat_template(...)` 的作用是：

- 把结构化 message 转成模型实际识别的 prompt 格式

如果不经过这一步，模型输入格式可能就不符合预训练/指令微调预期。

### 14.6 为什么要 `process_vision_info(messages)`

这是 Qwen-VL 多模态输入链路的一部分。

作用是：

- 从结构化消息中把图像/视频信息拆出来
- 交给 processor 去做视觉张量处理

### 14.7 为什么要裁掉 prompt token

`generate()` 的输出往往包含：

- 输入 prompt 对应 token
- 新生成 token

而服务端真正想返回的是“模型新生成的回答”，不是把 prompt 再复读一遍。

所以要做：

```python
out_ids[len(in_ids):]
```

这一步在生成式服务里非常常见。

## 16. 你必须理解的 trade-off

真正成熟的工程理解，不是知道“这能做什么”，而是知道“它为什么有代价”。

### 16.1 merge 的 trade-off

优点：

- 后处理更统一
- 部署更方便

代价：

- 模型体积更大
- 失去 LoRA 轻量表示的灵活性

### 16.2 prune 的 trade-off

优点：

- 有机会进入稀疏加速链
- 更贴近结构化部署优化

代价：

- 精度可能下降
- 不一定所有后端都真正吃到速度收益

### 16.3 quantize 的 trade-off

优点：

- 显存更小
- 推理更快
- 更适合部署

代价：

- 数值误差
- 精度下滑风险
- 校准策略敏感

### 16.4 TensorRT 的 trade-off

优点：

- 高性能
- 低延迟
- 更像生产级推理

代价：

- 环境复杂
- 参数复杂
- 调试成本高
- 不如 HF 生态灵活

### 16.5 API 协议适配的 trade-off

优点：

- 调用统一
- 业务系统接入方便
- 易于与现有客户端兼容

代价：

- 如果底层还是 HF generate，吞吐上限有限
- 真要极致性能，还得继续升级推理后端
